# 2026-03-11 実験記録
## iPadマスクエクスポート形式の修正（グレースケール→カラーRGBA）

### 概要 (Overview)
iPadから送信されるアノテーションマスクが真っ黒に見え、再インポート時にアノテーションが認識されない問題を修正。
原因はiPadのエクスポート形式（グレースケールPNG, ピクセル値=クラスID）とインポート側の期待形式（カラーRGBA PNG）の不一致。

### 変更元ファイル (Source file)
- `AnnottyHIL/ViewModels/CanvasViewModel.swift` — `exportMaskForServer()` 関数

### Before / After 比較
| 項目 | Before | After |
|------|--------|-------|
| 出力形式 | グレースケールPNG (mode=L) | カラーRGBA PNG |
| ピクセル表現 | クラスID直値 (0-9) → ほぼ黒 | クラス色 (R,G,B,A) → 視認可能 |
| 出力サイズ | 256×256 固定 | 512×512 固定 |
| 背景ピクセル | 0 (黒) | (0,0,0,0) 透明 |
| サーバー互換性 | reconstructor.pyの`mode=="L"`分岐で動作 | `RGBA`分岐で色マッチング動作 |
| iPad再インポート | `findNearestClassID`が(1,1,1)等にマッチ不可 | 正しいクラス色でマッチ成功 |

### 変更理由 (Reason for change)
1. **見た目の問題**: クラスID 1-9 はグレースケール値として 1-9/255 であり、人間には真っ黒にしか見えない
2. **ラウンドトリップ破綻**: iPadの`loadAnnotation()`はRGBA前提でクラス色ルックアップを行うため、グレースケールPNGを再読み込みすると全ピクセルが誤判定される
3. **サイズ不一致**: 元コードは256×256固定だが、大半のデータは512×512

### ハマった点と解決策 (Troubleshooting)
- `createColoredPNG()`（ローカル保存用）が既に同等のカラーRGBA変換を実装済みだったが、サーバー送信用の`exportMaskForServer()`は別実装でグレースケールのままだった
- 背景の扱い: `createColoredPNG`は白(255,255,255)、`exportMaskForServer`は透明(0,0,0,0)に統一。`loadAnnotation`の`isBackground`判定は両方カバー済み

### 残課題
- `trainer.py` はバイナリ（赤色のみ）検出のまま → マルチクラス対応が別途必要
