# 2026-03-11 実験記録
## iPadマスクエクスポート形式の修正（グレースケール→カラーRGBA）

### 概要 (Overview)
iPadから送信されるアノテーションマスクが真っ黒に見え、再インポート時にアノテーションが認識されない問題を修正。
原因はiPadのエクスポート形式（グレースケールPNG, ピクセル値=クラスID）とインポート側の期待形式（カラーRGBA PNG）の不一致。

### 変更元ファイル (Source file)
- `AnnottyHIL/ViewModels/CanvasViewModel.swift` — `exportMaskForServer()` 関数

### Before / After 比較
| 項目 | Before | After |
|------|--------|-------|
| 出力形式 | グレースケールPNG (mode=L) | カラーRGBA PNG |
| ピクセル表現 | クラスID直値 (0-9) → ほぼ黒 | クラス色 (R,G,B,A) → 視認可能 |
| 出力サイズ | 256×256 固定 | 512×512 固定 |
| 背景ピクセル | 0 (黒) | (0,0,0,0) 透明 |
| サーバー互換性 | reconstructor.pyの`mode=="L"`分岐で動作 | `RGBA`分岐で色マッチング動作 |
| iPad再インポート | `findNearestClassID`が(1,1,1)等にマッチ不可 | 正しいクラス色でマッチ成功 |

### 変更理由 (Reason for change)
1. **見た目の問題**: クラスID 1-9 はグレースケール値として 1-9/255 であり、人間には真っ黒にしか見えない
2. **ラウンドトリップ破綻**: iPadの`loadAnnotation()`はRGBA前提でクラス色ルックアップを行うため、グレースケールPNGを再読み込みすると全ピクセルが誤判定される
3. **サイズ不一致**: 元コードは256×256固定だが、大半のデータは512×512

### ハマった点と解決策 (Troubleshooting)
- `createColoredPNG()`（ローカル保存用）が既に同等のカラーRGBA変換を実装済みだったが、サーバー送信用の`exportMaskForServer()`は別実装でグレースケールのままだった
- 背景の扱い: `createColoredPNG`は白(255,255,255)、`exportMaskForServer`は透明(0,0,0,0)に統一。`loadAnnotation`の`isBackground`判定は両方カバー済み

### 残課題
- `trainer.py` はバイナリ（赤色のみ）検出のまま → マルチクラス対応が別途必要

## AI Predict（オンデバイス推論）の修正

### 概要 (Overview)
サーバーからダウンロードしたU-Netモデルで「AI predict」ボタンを押しても反応しない問題を修正。
サーバーモデル（1ch/256px/10クラス）とiPad推論コード（3ch/512px/バイナリ）の仕様不一致が原因。

### 変更元ファイル (Source file)
- `AnnottyHIL/Services/UNet/UNetService.swift` — 推論パイプライン
- `AnnottyHIL/ViewModels/CanvasViewModel.swift` — `applyUNetMask()` マスク適用
- `server/config.py` — モデルファイルパス

### Before / After 比較
| 項目 | Before | After |
|------|--------|-------|
| 入力チャンネル | 常に3ch RGB [1,3,512,512] | bundled: 3ch RGB / downloaded: 1ch gray [1,1,256,256] |
| 入力サイズ | 常に512×512 | bundled: 512 / downloaded: 256 |
| 出力処理 | sigmoid → threshold 0.5 → バイナリ(0/1) | bundled: バイナリ / downloaded: argmax → クラスID(0-9) |
| マスク適用 | 非ゼロ→currentClassID固定 | バイナリ: currentClassID / マルチクラス: クラスIDそのまま |
| config BEST_MODEL_PATH | `best.pt` (存在しない) | `best.pth` (実ファイル名) |

### 変更理由 (Reason for change)
1. **入力形式不一致**: サーバーモデルは `VanillaUNet(in_channels=1)` でグレースケール1ch入力だが、iPadは3ch RGB MLMultiArrayを作成していた → CoreML推論が形状エラーで失敗
2. **サイズ不一致**: サーバーの `IMAGE_SIZE=256` に対しiPadは512固定
3. **出力形式不一致**: 10クラスlogits出力に対しバイナリsigmoid処理をしていた
4. **ファイル名不一致**: `best.pt` vs `best.pth`

### ハマった点と解決策 (Troubleshooting)
- bundledモデル（4-fold ensemble, RGB, 512px）とdownloadedモデル（single, gray, 256px）で完全に異なるパイプラインが必要
- `predictMulticlass()` と `predictBundledEnsemble()` に分離して対応
- マルチクラスマスクのアップスケールはnearest neighbor必須（bilinearだとクラスIDが補間されて壊れる）
- `applyUNetMask()` で `maxVal > 1` でマルチクラス/バイナリを自動判定
