## Annotty-MRI: クラス定義UI + label_config.json 双方向同期の実装

### 概要 (Overview)
annotty-hilをフォークし、医用画像（眼窩MRI）セグメンテーション向けにクラス定義UIを改修。
- 8色常時表示 → 定義済みクラスのみ表示 + 「Add Class」ボタン
- label_config.json による PC ↔ iPad 双方向クラス定義同期
- クラス未定義時の描画ブロック（医用画像の安全策）

### 変更元ファイル (Source file)
- `AnnottyHIL/Services/FileManager/ProjectFileService.swift`
- `AnnottyHIL/ViewModels/CanvasViewModel.swift`
- `AnnottyHIL/Views/RightPanel/RightPanelView.swift`
- `AnnottyHIL/Views/RightPanel/ImageSettingsOverlayView.swift`
- `AnnottyHIL/Views/MainView.swift`

### Before / After 比較
| 項目 | Before | After |
|------|--------|-------|
| クラス表示 | 8色全て常時表示 | 定義済みのみ + Add/Removeボタン |
| クラス名管理 | UserDefaultsのみ | UserDefaults + label_config.json |
| クラス定義の起点 | iPad手動入力のみ | PC(JSON同梱) or iPad(Add Class) |
| 描画制御 | 常に描画可能 | クラス未定義時は描画ブロック |
| Settings画面 | 8クラス分TextField | 定義済み分のみ |

### 変更理由 (Reason for change)
- 医用画像で名前未定義の色マスクがサーバーに戻るリスクを排除
- PC側でラベル定義を事前準備 → iPadで即座に正しいクラス名で作業開始
- iPad側でクラス定義 → PCに戻る際にJSONが自動生成される双方向性

### ハマった点と解決策 (Troubleshooting)
- ディレクトリリネーム時にプロセスがフォルダを掴んでいてmvできず → cpで対応、元フォルダは手動削除

## DICOM確認 + NIfTI変換

### 解析手法 (Methods)
- 対象データ: `T2_Coronal/*.dcm` (291ファイル)
- 使用ツール: pydicom, SimpleITK, nibabel, matplotlib
- 環境: `uv venv .venv --python 3.11`

### 変更元ファイル (Source file)
- `scripts/view_dicom.py` (新規) — DICOMメタデータ確認・一覧画像生成

### ハマった点と解決策 (Troubleshooting)
- 291 DCMが1ディレクトリに混在 → pydicomでPatientID/SeriesInstanceUIDごとに分類して対処
- 画像サイズが異なる（512×512 vs 256×256） → np.stackできず、シリーズ単位で処理するよう修正
- SimpleITKのスライスソート → ImagePositionPatientからスライス法線方向を計算して射影ソート
- T2 FLAIRは信号特性が異なるため除外（sub-006の2016-10-03） → 最終10ボリューム

## サーバーAPI改修 + NIfTI再統合モジュール

### 概要 (Overview)
サーバーを眼底PNG用フラット構造から、MRI症例ベース（`slices/{case_id}/`）構造に改修。
マスクPNG → NIfTIラベルマップ再統合モジュールを新規作成。

### 変更元ファイル (Source file)
- `server/config.py` — MRIパス（NIFTI_RAW_DIR, NIFTI_LABELS_DIR, SLICES_DIR）追加、NUM_CLASSES=7
- `server/data_manager.py` — 症例ベースに全面書き換え
- `server/main.py` — 症例ベースAPI（/cases/...）に全面書き換え
- `server/medical_adapter/slicer.py` (新規) — NIfTI → スライスPNG + manifest生成
- `server/medical_adapter/reconstructor.py` (新規) — マスクPNG → NIfTIラベルマップ再統合

### Before / After 比較

| 項目 | Before | After |
|------|--------|-------|
| データ構造 | `images_completed/` + `images_unannotated/` フラット | `slices/{case_id}/images/` 症例ベース |
| API設計 | `GET /images`, `PUT /submit/{id}` | `GET /cases`, `GET /cases/{case_id}/images`, `PUT /cases/{case_id}/submit/{id}` |
| 画像形式 | RGB PNG（眼底） | W/L適用済みRGB PNG（MRIスライス） |
| マスク形式 | バイナリ赤色PNG (NUM_CLASSES=1) | インデックスカラーPNG (NUM_CLASSES=7) |
| 座標管理 | なし | slice_manifest.json（affine, slice_index） |
| 再統合 | なし | マスクPNG → _dseg.nii.gz + JSON sidecar |

### 新規APIエンドポイント

| Method | Path | 説明 |
|--------|------|------|
| GET | `/cases` | 症例一覧 |
| GET | `/cases/{case_id}/images` | スライス一覧 |
| GET | `/cases/{case_id}/images/{id}/download` | スライスPNGダウンロード |
| PUT | `/cases/{case_id}/submit/{id}` | マスクPNGアップロード |
| GET | `/cases/{case_id}/labels/{id}/download` | マスクダウンロード |
| GET | `/cases/{case_id}/next` | 次の未ラベルスライス |
| GET | `/cases/{case_id}/manifest` | slice_manifest取得 |
| GET | `/cases/{case_id}/label_config` | クラス定義取得 |
| POST | `/cases/{case_id}/reconstruct` | NIfTI再統合実行 |

### テスト結果
- DataManager: 10症例・213スライス正常認識
- reconstructor: ダミーRGBAマスク（赤=class1, 緑=class4）→ NIfTI再統合成功
  - unique labels [0, 1, 4] 正常、affine一致確認済み
